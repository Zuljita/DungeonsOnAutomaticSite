User manual

Run a dungeon before the table gets restless.

This manual is the public guide for Dungeons on Automatic. It explains the workflow, what each control means, how to export useful files, and how to recover when a seed has developed professional objections.

Quick start

  1. Choose a dungeon size, biome mix, party CER, challenge profile, generosity, door security, and travel distance.
  2. Select the source books your table actually uses, then apply the staged source changes before generating.
  3. Add a Session Brief if you want a theme, tone, set-piece request, danger hint, or party spotlight.
  4. Generate a seed. Review the GM map, player map, and room list.
  5. Lock any rooms you want to preserve, then reroll specific details as needed.
  6. Export the map and room key into your prep tool or VTT.
Best first run: Standard size, 4 biomes, Average challenge, 100% generosity, Standard door security, and the default required sources. Raise danger after you know the party's appetite for terrible architectural decisions.

Source-book filtering

Required sources stay locked on so the generator has the structural data it needs. Monster books can be bulk enabled or disabled, and source changes are staged until you choose Apply or Generate.

Profiles and persistence

Generation profiles can be exported and imported. The app also persists your last settings locally so a table setup survives a restart.

Session Brief and Dungeon Intent

Before generation, the Session Brief can capture tonight's theme, factions, tone, set pieces, danger target, and skills or advantages you want to spotlight. You can fill it in directly; with a local LLM connected, Suggest chips and the optional Dungeon Interview can turn loose notes into structured brief fields.

After generation, Dungeon Intent gives you finer control: brief Markdown, global tag bias, region-specific pressure, and set-piece requests. Room-level course correction can reroll encounters, treasure, or a zone with optional tag bias.

The AI serves as a director. Suggestions become typed tool calls, deterministic generation passes, or schema-checked prose. Source-filtered DFRPG data and validators remain authoritative.

Maps and room keys

The GM map contains everything: room numbers, secret doors, hazards, keys, encounters, markers, GM lights, treasure hooks, and notes. The player map removes hidden or GM-only information so it can be safely shared during play.

Map controls

The map view includes biome underlay and grid toggles, pan and zoom controls, and zoom-to-fit for returning to the full dungeon view.

Room keys

Each room entry is meant to be used at the table. It should include room purpose, visible features, encounter pressure, treasure hooks or scaled treasure notes, hazards, doors, and tags that explain why the room belongs where it does.

Player-safe behavior

Player exports hide secret or GM-only details, room labels, GM lights, markers, encounters, traps, treasure hooks, and notes where applicable.

VTT precision

Foundry scene JSON is available today. Universal VTT exports include embedded map images, walls, portals, lights, and GM/player modes. Importer hardening remains active roadmap work.

Stage Manager

The Stage Manager is the in-character director layer and the real progress/event surface. Kovath, Zarlzazz, Vorga, and Mizzik narrate what the generator is doing while generation, validation, exports, and polish work happen underneath.

Local AI direction and narrative polish

Local AI is optional. Connect Ollama, LM Studio, or llama.cpp in Settings to enable brief suggestions, the Dungeon Interview, director actions, and narrative polish. The desktop app talks to the provider running on your own machine.

LLM director actions use a constrained list of validated tool calls. Source filters and structural validators review every proposed DungeonState change.

Current limitations

  • Generation is single-level today.
  • Stairs can be placed and exported. Multi-floor graphing is future work.
  • Universal VTT importer QA is ongoing.
  • Custom treasure per room and custom container tables remain roadmap work.
  • Hallways currently use fixed widths.

Troubleshooting

The dungeon feels too lethal

Lower challenge, reduce peril-heavy biomes, or reroll encounters while keeping the room layout locked.

The map feels too linear

Increase size, raise branch/dead-end tolerance, or add a second biome to force more internal transitions.

The VTT import is misaligned

Confirm the target grid size, export scale, and whether the VTT expects pixels per square or square size in scene units.

I need to report a bug

Keep the seed, settings, app version, export format, and the room number or map area where the problem appears. That is the incident report. The staff will pretend to appreciate it.
